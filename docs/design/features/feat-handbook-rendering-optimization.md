# Handbook 渲染性能優化 - 增量優化方案

## 概述

本文檔針對 `backend/src/server/routes/handbooks.ts` 中的 handbook 渲染流水線提出增量優化方案。目標是在保持代碼可維護性和最小化風險的前提下，將渲染性能提升 30-40%。

**狀態：** 設計階段
**預計實施時間：** 2-3 天
**預期提升：** 渲染速度提升 30-40%，CPU 負載降低

---

## 當前性能問題

### 1. 重複的表達式 ID 計算
當前代碼對同一個表達式多次調用 `db.stableExpressionId()`：
- 初始標籤提取階段（line 125）
- 標題渲染時（line 358）
- 內容佔位符替換時（line 470）

由於 `stableExpressionId()` 需要對文本和語言代碼進行哈希計算，每個表達式調用 3 次會造成大量 CPU 浪費。

### 2. 重複的音頻 URL 解析
存儲在 JSON 格式中的音頻 URL 在每次 `renderItem()` 調用時都會被解析：
- 第 341-343 行：爲標題解析 JSON 並提取 URL
- 第 365-369 行：爲內容再次解析 JSON 並提取 URL
- 第 452-456 行：佔位符替換時第三次解析 JSON 並提取 URL

這對 handbook 中的每個表達式都會發生，儘管音頻 URL 在單次渲染操作中永遠不會改變。

### 3. 缺少內存緩存
每次 handbook 渲染時：
- 查詢數據庫獲取所有表達式（line 151）
- 查詢數據庫獲取表達式組
- 沒有緩存跨請求中頻繁使用的表達式

一個包含 50 個唯一表達式的 handbook 每次渲染會執行 100+ 次數據庫查詢。

### 4. 低效的 HTML 生成
`renderItem()` 函數重複構建相同的 HTML 結構：
- 相同的語言顏色生成邏輯
- 相同的翻譯組格式化
- 相同的行內 onclick 處理器

這產生了不必要的字符串操作開銷。

---

## 優化策略

### 策略 1：在標籤提取階段緩存表達式 ID
**目標：** 消除重複的 `stableExpressionId()` 調用

**實施方案：**
1. 在初始標籤提取階段創建一個 `Map<string, number>` 來緩存表達式 ID（第 92 行）
2. 在發現時存儲計算出的 ID：`expressionIdCache.set(key, id)`，其中 `key = "${text}|${lang}"`
3. 用緩存查找替換後續的 `db.stableExpressionId()` 調用

**代碼影響：**
- 修改 `renderHandbookInternal()` 函數
- 在標籤提取循環前添加緩存 map 初始化
- 替換第 358、470 行的調用爲緩存查找

**預期收益：** 哈希計算開銷減少 20-25%

### 策略 2：預處理音頻 URL
**目標：** 每個表達式只解析一次音頻 URL，而非每次渲染都解析

**實施方案：**
1. 獲取表達式組後（第 151 行），立即創建音頻 URL 緩存 map
2. 遍歷 `expressionMap` 並提取音頻 URL
3. 存儲在 `Map<number, string>` 中，映射 `expressionId → audioUrl`
4. 修改 `renderItem()` 接受預計算音頻 URL 而不是表達式對象

**代碼影響：**
- 在第 175 行後添加音頻 URL 提取循環
- 將 `renderItem()` 籤名從 `(id, text, audioUrl, isTitle, meaningId)` 改爲 `(id, text, audioUrl, isTitle, meaningId, audioCache?)`
- 從 `renderItem()` 中移除 JSON 解析

**預期收益：** JSON 解析開銷減少 10-15%

### 策略 3：添加表達式 LRU 緩存
**目標：** 減少頻繁使用表達式的數據庫查詢

**實施方案：**
1. 添加表達式組的內存 LRU 緩存（最多 1000 條，10 分鐘 TTL）
2. 調用 `db.getExpressionsGroups()` 前檢查緩存
3. 緩存鍵：`expressionId_hash`，hash 爲逗號分隔的排序後的表達式 ID
4. 在表達式更新時使緩存失效（已通過 `clearStatisticsCache()` 實現）

**代碼影響：**
- 在 `handbooks.ts` 模塊級別添加緩存
- 在第 151 行前添加緩存檢查
- 在第 164 行後添加緩存寫入

**預期收益：** 流行 handbooks 的數據庫查詢減少 30-50%

### 策略 4：優化 HTML 生成
**目標：** 減少字符串操作開銷

**實施方案：**
1. 在渲染前預計算所有目標語言的語言顏色
2. 將通用 HTML 生成邏輯提取到輔助函數
3. 高效使用模板字符串（避免嵌套字符串拼接）

**代碼影響：**
- 添加 `precomputeLanguageColors()` 輔助函數
- 重構 `renderItem()` 使用預計算的顏色
- 將翻譯組渲染提取到單獨的函數

**預期收益：** HTML 生成開銷減少 5-10%

## 性能指標

### 基準（當前）
- 表達式 ID 計算：約 50-100 個表達式 × 3 次調用 = 150-300 次哈希操作
- 音頻 URL 解析：約 50-100 個表達式 × 3 次解析 = 150-300 次 JSON 解析
- 數據庫查詢：每次渲染 2 次查詢 + 每個唯一表達式組 1 次
- 總渲染時間：約 500-2000ms（取決於 handbook 大小）

### 目標（優化後）
- 表達式 ID 計算：約 50-100 個表達式 × 1 次調用 = 50-100 次哈希操作（減少 66%）
- 音頻 URL 解析：約 50-100 個表達式 × 1 次解析 = 50-100 次 JSON 解析（減少 66%）
- 數據庫查詢：每次渲染 2 次查詢 + 表達式緩存命中（減少 30-50%）
- 總渲染時間：約 300-1200ms（提升 30-40%）

## 風險評估

### 低風險
- 表達式 ID 緩存：純算法改進，無狀態變更
- 音頻 URL 預處理：隔離變更，易於測試

### 中等風險
- LRU 緩存：引入內存狀態，必須確保緩存失效正確工作
- HTML 重構：潛在的回歸風險，需要全面測試

### 緩解措施
- 爲每個優化添加全面的單元測試
- 監控生產環境中的緩存命中率
- 如果出現問題，添加功能標誌禁用緩存
- 每個階段前後進行性能基準測試

---

## 測試策略

### 單元測試
- 測試表達式 ID 緩存正確性
- 測試音頻 URL 提取和緩存
- 測試 LRU 緩存行爲（大小、TTL、驅逐）
- 測試 HTML 生成輔助函數

### 集成測試
- 測試不同大小 handbook 的完整渲染流水線
- 測試 handbook/表達式更新時的緩存失效
- 測試並發渲染請求

### 性能測試
- 每個優化前後基準測試渲染時間
- 測量渲染期間的 CPU 使用率
- 跟蹤數據庫查詢計數
- 監控內存使用（LRU 緩存影響）

## 回滾計劃

每個優化都是獨立的，可以單獨回滾：

1. **表達式 ID 緩存：** 恢復直接調用 `db.stableExpressionId()`
2. **音頻 URL 預處理：** 在 `renderItem()` 中恢復 JSON 解析
3. **LRU 緩存：** 添加功能標誌禁用緩存
4. **HTML 優化：** 恢復到原始 HTML 生成代碼

不需要數據庫架構變更，因此不需要遷移回滾。
