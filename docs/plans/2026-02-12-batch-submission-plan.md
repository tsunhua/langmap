# 开发计划：批量语义提交功能实现 (Batch Submission Implementation Plan)

**日期**：2026-02-12
**状态**：待评审
**目标**：实现跨语言词句的批量提交、智能语义锚点分组以及词句更正时的级联更新机制。

## 1. 任务概述

本功能旨在提升语料录入效率，通过一次性提交多语言同义词句，并由后端自动化处理语义关联（Self-referencing `meaning_id`）。核心挑战在于确定性 ID 的查重效率、复杂的锚点选择逻辑以及跨表的数据一致性维护。

## 2. 实施阶段

### 阶段 1：后端开发 (Backend Implementation)
重点在于 API 逻辑和数据库原子操作的实现。

- **1.1 核心算法实现**：
    - ✅ **已实现**：基于 `text + language_code` 的唯一 ID 生成函数 (`stableExpressionId`)。
    - **实现智能语义锚点选择器**：按以下步骤（批量查重 → 排序 → 选择）确定 `meaning_id`：
        1. 基于输入内容计算所有 IDs，执行**单次批量 SQL 查重**。
        2. 按语言权重对输入列表进行预排序。
        3. 遍历排序后的列表，匹配批量查询结果，优先提取已存在的 `meaning_id`。
        4. 若均无关联，则取排序后首条记录的 `expression_id` 作为锚点。
- **1.2 批量提交 API (`POST /api/v1/expressions/batch`)**：
    - 支持 `IN` 子句的批量 ID 查重。
    - 使用 `INSERT ... ON CONFLICT(id) DO UPDATE` (UPSERT) 实现原子化的记录创建与更新。
    - 确保操作在 D1 事务中执行。
- **1.3 词句修正 API (`PATCH /api/v1/expressions/:id`)**：
    - 实现级联更新逻辑：当锚点 ID 变更时，更新所有引用旧 ID 的 `meaning_id`。
    - 实现资产迁移：迁移 `collection_items` 的关联。
    - 实现版本管理：将旧记录快照存入 `expression_versions`。

### 阶段 2：前端开发 (Frontend Implementation)
重点在于提供极简且高效的多语言录入体验。

- **2.1 UI 组件设计**：
    - 开发“批量录入”表单，支持动态增加语言行。
    - 适配地图选择器（Region）与标签输入。
- **2.2 数据交互与状态管理**：
    - 集成后端批量 API。
    - 实现提交进度反馈与详细的结果报告（展示哪些是新建，哪些是更新）。

### 阶段 3：测试与验证 (Verification)
- **单元测试**：验证锚点选择逻辑在各种排列组合下的正确性。
- **并发测试**：确保在 D1 环境下，高并发提交不会导致语义组分裂。
- **端到端回归**：验证修正词句后，地图各点及用户收藏夹是否同步更新。
