# 開發計劃：批量語義提交功能實現 (Batch Submission Implementation Plan)

**日期**：2026-02-12
**狀態**：待評審
**目標**：實現跨語言詞句的批量提交、智能語義錨點分組以及詞句更正時的級聯更新機制。

## 1. 任務概述

本功能旨在提升語料錄入效率，通過一次性提交多語言同義詞句，並由後端自動化處理語義關聯（Self-referencing `meaning_id`）。核心挑戰在於確定性 ID 的查重效率、複雜的錨點選擇邏輯以及跨表的數據一致性維護。

## 2. 實施階段

### 階段 1：後端開發 (Backend Implementation)
重點在於 API 邏輯和數據庫原子操作的實現。

- **1.1 核心算法實現**：
    - ✅ **已實現**：基於 `text + language_code` 的唯一 ID 生成函數 (`stableExpressionId`)。
    - **實現智能語義錨點選擇器**：按以下步驟（批量查重 → 排序 → 選擇）確定 `meaning_id`：
        1. 基於輸入內容計算所有 IDs，執行**單次批量 SQL 查重**。
        2. 按語言權重對輸入列表進行預排序。
        3. 遍歷排序後的列表，匹配批量查詢結果，優先提取已存在的 `meaning_id`。
        4. 若均無關聯，則取排序後首條記錄的 `expression_id` 作爲錨點。
- **1.2 批量提交 API (`POST /api/v1/expressions/batch`)**：
    - 支持 `IN` 子句的批量 ID 查重。
    - 使用 `INSERT ... ON CONFLICT(id) DO UPDATE` (UPSERT) 實現原子化的記錄創建與更新。
    - 確保操作在 D1 事務中執行。
- **1.3 詞句修正 API (`PATCH /api/v1/expressions/:id`)**：
    - 實現級聯更新邏輯：當錨點 ID 變更時，更新所有引用舊 ID 的 `meaning_id`。
    - 實現資產遷移：遷移 `collection_items` 的關聯。
    - 實現版本管理：將舊記錄快照存入 `expression_versions`。

### 階段 2：前端開發 (Frontend Implementation)
重點在於提供極簡且高效的多語言錄入體驗。

- **2.1 UI 組件設計**：
    - 開發「批量錄入」表單，支持動態增加語言行。
    - 適配地圖選擇器（Region）與標籤輸入。
- **2.2 數據交互與狀態管理**：
    - 集成後端批量 API。
    - 實現提交進度反饋與詳細的結果報告（展示哪些是新建，哪些是更新）。

### 階段 3：測試與驗證 (Verification)
- **單元測試**：驗證錨點選擇邏輯在各種排列組合下的正確性。
- **並發測試**：確保在 D1 環境下，高並發提交不會導致語義組分裂。
- **端到端回歸**：驗證修正詞句後，地圖各點及用戶收藏夾是否同步更新。

## 3. 關鍵裏程碑 (Milestones)

| 裏程碑 | 交付物 | 預計耗時 |
| :--- | :--- | :--- |
| **M1: 後端核心完成** | 支持 UPSERT 和 智能錨點的 Batch API | 3 天 |
| **M2: 級聯處理完成** | 可靠的詞句更正與數據遷移邏輯 | 2 天 |
| **M3: 前端原型交付** | 可交互的批量錄入界面 | 3 天 |
